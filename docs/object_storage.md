
### Как организовано разделение при хранении:

- Ключ (key) объектов в Object Storage формируется так:
  ```
  videos/{orgID}/{videoID}/{fileName}
  ```
  Здесь `orgID` — идентификатор организации, `videoID` — уникальный ID видео, `fileName` — имя файла.

- Таким образом, все видео одной организации хранятся под своим префиксом `videos/{orgID}`, что обеспечивает логическое разделение данных разных организаций.

- В коде upload и download операций используется именно такой ключ, благодаря чему файлы пользователей из разных организаций распределены по разным "папкам" внутри бакета.

- Кроме того, при доступе к видео всегда проверяется, что пользователь принадлежит указанной организации (`if video.OrgID != orgID { return error }`), что обеспечивает контроль доступа на уровне backend.

### Итог

Данные в Object Storage организованы с разделением по организациям через соответствующие префиксы ключей. Так обеспечивается изоляция данных разных организаций в одном S3-совместимом бакете.


- **videoID** — это уникальный идентификатор видео в базе данных. Он создаётся автоматически (например, с помощью UUID) и служит главным ключом для однозначной идентификации видео в системе. Видео найти или оперировать можно именно по этому ID.

- **fileName** — это имя файла видео, например, "video.mp4". Это имя файла, под которым видео сохраняется и отображается пользователю. Имя может быть произвольным и не уникальным по системе, но хранится для удобства отображения и поиска по названию.

При хранении на Object Storage ключ формируется как:
```
videos/{orgID}/{videoID}/{fileName}
```
То есть `videoID` гарантирует уникальность и однозначность объекта в системе, а `fileName` — это человекочитаемое имя файла для удобства.

Таким образом, `videoID` — технический уникальный идентификатор, а `fileName` — имя файла для отображения и поиска.
