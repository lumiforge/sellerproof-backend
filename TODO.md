# Цель
Нужно изменить проект так чтобы для трех разных подписок free, pro и enterprise видео сохранялось только одно значение на разные бакеты.
free на sub12-ice
pro на sub24-ice
enterprise на sub36-ice

Миграций никаких не требуется, так как сервис не запущен можно менять БД изменяя метод internal/ydb/client.go:82-82```createTables```. 

Изменить код проекта и Makefile для сохранения значения переменных.
# Детали

1.  **Логика Публикации (Public Bucket):**
    В текущем коде (`internal/video/service.go:PublishVideo`) при публикации видео оно **копируется** из приватного бакета в отдельный публичный бакет (`SPObjStorePublicBucket`).
    *   **Пояснение:** убрать эту логику,  публичные видео тоже должны храниться в этих новых бакетах (`sub12-ice`, `sub24-ice`, `sub36-ice`);
    
2.  **Изменение схемы БД (Таблица `videos`):**
    Так как пользователь может сменить тариф (например, с Free на Pro), нам нельзя полагаться только на *текущую* подписку пользователя для определения бакета, где лежит старое видео.
    *   **Пояснение:** добавь колонку `bucket_name` (Text) в таблицу `videos` в файле `internal/ydb/client.go`. При загрузке видео мы будем записывать туда имя бакета, соответствующее текущему тарифу. При скачивании/удалении будем брать имя бакета из этой колонки.

3.  **Переменные окружения:**
    Заменить переменную `SP_OBJSTORE_PRIVATE_BUCKET` на три новые.
    *   **Пояснение:** Назвать их `SP_OBJSTORE_BUCKET_FREE`, `SP_OBJSTORE_BUCKET_PRO`, `SP_OBJSTORE_BUCKET_ENTERPRISE`.

4.  **Обработка неизвестных планов (Fallback):**
    В коде есть инициализация планов (`free`, `pro`, `enterprise`).
    *   **Пояснение:** Если в будущем появится новый план (или при какой-то ошибке план не определится), сохранять видео по умолчанию в `sub12-ice` как самый дешевый/базовый;

5.  **Storage Class (Класс хранилища):**
    Названия бакетов содержат суффикс `-ice` (обычно это Cold Storage / Ice в терминологии Yandex Cloud).
    *   **пояснение:** не нужно явно указывать `StorageClass` при загрузке через AWS SDK,  эти бакеты уже настроены на уровне облака так, что все объекты в них автоматически попадают в нужный класс хранилища;


**Важно:** это production-ready решение, это **не** MVP, писать только production-ready код.
